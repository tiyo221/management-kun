## プロジェクト概要

**マネジメントくん** — 既存の5つの単一HTMLツール（目標みえるくん / スキル可視化 / タスク可視化 / todo-kun / WBS）を、**共通デザイン・共通データ規格・共通マスタ**のもとに1つへ統合した管理ツール。

- 主要ファイル: [`index.html`](index.html)（シェル。これを開く）＋ `shared/*`（共有資産）＋ `modules/*`（各モジュール）
- 仕様書: [`spec.md`](spec.md) — アーキテクチャ・共通データ規格・名寄せ・デザイン統一・MVP/フェーズの定義（**本書より spec.md を優先**）
- デザイン定義: [`DESIGN.md`](DESIGN.md) — Notion 風デザインシステム（トークン・コンポーネント）
- 規約・チェックリスト: [`CONVENTIONS.md`](CONVENTIONS.md) — UI/レイアウト規約・モジュール分割（logic/view）・完成チェックリスト（**モジュール追加/修正時は必ず参照・自己点検**）
- データ永続化: ブラウザの `localStorage`（`mk:` プレフィックスで統一）。外部送信は行わない。
- 共有・バックアップ・AI 保守: JSON エクスポート / インポート（全体 / モジュール / 人 / プロジェクト単位）。skills / wbs は CSV にも対応。

目的（詳細は [`spec.md`](spec.md) §1）:

- 5ツールのデザインを統一する（[`DESIGN.md`](DESIGN.md) の同一トークン・コンポーネントを全モジュールで使用）
- JSON / localStorage / CSV のデータ規格を1本化する
- 「メンバー（人）」「プロジェクト」を全モジュール横断のマスタとして共有する
- 共通部分を共有資産に切り出し、モジュールごとの重複実装をなくす（保守性）

設計思想（重要）: **ビルド不要・外部依存ゼロ・`file://` 動作・ローカル完結**を崩さない。
ただし単一ファイルではなく **「単一ページ・シェル ＋ 共有資産 ＋ モジュール」** 構成とする（保守性のためファイルは分割するが、`file://` で確実に動かすため **ES Modules / `fetch()` は使わず**、classic `<script src>` + グローバル名前空間 `window.MK` で連携する。詳細は [`spec.md`](spec.md) §3）。画像はインライン SVG か絵文字、グラフは外部ライブラリを使わず Canvas / インライン SVG で自前描画、フォントはシステムフォントスタック（[`DESIGN.md`](DESIGN.md) の Notion Sans は外部読込しない）。

## 構成

ディレクトリ構成は [`spec.md`](spec.md) §3.2 を正とする。

```
index.html              … シェル（ランチャー兼ホスト）。モジュールを同一ページ内に差し替えロード。
shared/
  design.css            … DESIGN.md 由来のトークン + 共通コンポーネント + レスポンシブ
  core.js               … window.MK 名前空間・共通ユーティリティ（el/uid/日付/名寄せキー/escape）
  store.js              … localStorage 永続化・スキーマ/マイグレーション
  people.js             … 人の管理（members）の CRUD・名寄せ解決API
  projects.js           … プロジェクト管理（projects）の CRUD・名寄せ解決API
  io.js                 … JSON エンベロープ入出力・CSV パーサ/シリアライザ
  ui.js                 … 共通UI部品（レイアウト/フォーム/モーダル/トースト）
  sample.js             … サンプルデータ投入
modules/<id>/
  logic.js              … データ・計算・CRUD（DOM非依存）  ※<id> = goals/skills/workload/todo/wbs
  view.js               … 描画・イベント（MK.registerModule）
```

- 各モジュールは `MK.registerModule(id, def)` で自身を登録し、シェルが `def.mount(container, ctx)` を呼ぶ（API 契約は [`spec.md`](spec.md) §3.5）。
- **logic/view を必ず分割する**。logic は DOM に触れずデータ・計算を担い（`MK.logic[id]`）、view は描画のみ。共有ヘルパ・追加手順・完成チェックリストは [`CONVENTIONS.md`](CONVENTIONS.md)。
- モジュールは**自分の名前空間 `mk:module:<id>` のみ**書き込む。マスタの変更は必ず `ctx.people` / `ctx.projects` 経由で行う（直接 localStorage を触らない）。

### データ規格（共通）

- **localStorage キー**: `mk:people:v1` / `mk:projects:v1` / `mk:module:<id>:v1` / `mk:settings:v1`（[`spec.md`](spec.md) §4.1）
- **JSON エンベロープ**: `{ schema, schemaVersion, exportedAt, scope, people, projects, modules }`（§4.2）
- **共通マスタ**: People（人）と Project は**独立ドメイン**（別ストア・別管理画面）。モジュールは `memberId` / `projectId` で参照（§4.4 / §4.7）
- **名寄せ**: 名前（文字列）→マスタへ解決する共通フロー（§8）。MVP は自動作成のみ（レビュー画面は Phase 2）
- **スキーマ規約（AI 保守のため）**: 日時は ISO 8601 UTC 文字列、日付のみは `YYYY-MM-DD`。未設定の任意項目も省略せず `null` / `[]` / `""` を入れる。Import は「置換 / マージ」を選べる（マージは `id` 一致で上書き・なければ追加）
- **CSV**: UTF-8（読込は BOM 許容・出力は BOM 付き）・RFC 4180 準拠の自前パーサ・メンバー/プロジェクトは名前参照（§4.6）
- **堅牢性**: 壊れた localStorage / 不正な JSON でも起動し、エラーを握りつぶさず案内する（§10.1）

### モジュール

| id | 旧ツール | localStorage | CSV |
|---|---|---|---|
| goals | 目標みえるくん | `mk:module:goals:v1` | – |
| skills | スキル可視化 | `mk:module:skills:v1` | ✓ |

## コーディング規約

### 言語・実行環境
- Vanilla JS（ES2020+）。フレームワーク・トランスパイラ・バンドラを使わない。実行時に npm パッケージを含めない。
- ES Modules（`import` / `type="module"`）と `fetch()` は使わない（`file://` のため）。公開シンボルは `window.MK` 名前空間へ集約する。
- ブラウザ標準 API のみ。依存を足したくなったら、まず自前実装か不要化を検討する（原則・依存ゼロ）。

### モダン JS（古い書き方の抑制）
- **`var` を使わない。** 変数は `const` を既定とし、再代入が必要なときだけ `let` を使う。
- 比較は `===` / `!==`（`==` / `!=` を使わない）。
- 文字列結合はテンプレートリテラル、関数はアロー関数や分割代入など読みやすい現代的な書き方を優先する（ただし既存コードの流儀と一貫させる）。
- DOM はイベントは `addEventListener` で登録し、HTML 属性の `onclick=` 直書きや `document.write` を使わない。
- 反復は `for...of` / 配列メソッド（`map`/`filter`/`forEach` 等）を基本とする（WBS の階層走査などインデックス操作が自然な箇所は無理に置き換えない）。

### 構造・命名
- 共有ロジックは `shared/` に置き、モジュール間で重複させない。ただし共有化は **実際に2か所以上で必要になってから**（先回りの抽象化を避ける）。
- モジュールは自分の名前空間 `mk:module:<id>` のみ書き込み、マスタは `ctx.people` / `ctx.projects` 経由で触る（直接 localStorage を読み書きしない・[`spec.md`](spec.md) §3.5）。
- 色 / 余白 / 角丸 / タイポは [`DESIGN.md`](DESIGN.md) のトークン（CSS変数）経由で指定し、値の直書きをしない（§6.1）。
- ID 採番・JSON 入出力・CSV・名寄せは `shared/` の共通実装を使い、各モジュールで再実装しない。

### 安全・堅牢性
- ユーザー入力・取込データ（名前・タイトル・メモ等）を DOM に出すときは `textContent` を使うか必ずエスケープする。`innerHTML` に未エスケープ文字列を渡さない（**XSS 防止**。JSON / CSV 取込があるため重要）。
- localStorage は名前空間ごとに try/parse し、壊れても他へ波及させない。エラーは握りつぶさず案内する（§10.1）。
- 日時は ISO 8601 UTC、日付は `YYYY-MM-DD`。未設定の任意項目は `null` / `[]` / `""` で埋める。

### スタイル
- 既存コードの記法・命名・コメント密度に合わせる。新しい流儀を持ち込まない。コメントは日本語。

## オーバーエンジニアリングの防止

方針: 「**まずは MVP、気になったら作り直す**」。未決事項は最もシンプルな選択で確定する。

- **YAGNI**: いま必要な機能だけ作る。[`spec.md`](spec.md) §9 で Phase 2 / 将来に分類した項目（名寄せレビュー画面・マスタの事後マージ・プロジェクトダッシュボード・クロス機能）を MVP に先取りしない。
- **抽象化は後出し**: 設定オプション・汎用エンジン・プラグイン機構を「将来のため」に作らない。`registerModule` 以上の拡張ポイントを増やさない。共通化は2か所以上で必要になってから（rule of three）。
- **最小の選択**: 仕様が曖昧な点は最も単純な実装を選び、その判断を一言メモして進む。凝った可変機構を作らない。
- **決定の蒸し返し禁止**: spec が「やらない」と決めたこと（タスクの統一スキーマ化など §5.6）を再燃させない。新エンティティ / 新永続化キーの追加は spec 更新とセットで行う。
- **巨大化対策**: ファイルが肥大したら機能で分割するが、分割のための分割はしない。

迷ったら「これは MVP に要るか？ spec のどこに根拠があるか？」を先に確認する。

## MVP スコープ（まずこれを作る）

「まずは MVP、気になったら作り直す」方針。詳細は [`spec.md`](spec.md) §9。

- 統合 MVP は **基盤（シェル / デザイン統一 / 共通データ層 / 人・プロジェクトマスタ）＋ 5モジュール移植 ＋ 旧データ自動移行 ＋ 全体 JSON 入出力**まで。
- 移植は既存機能の維持＋デザイン/データ規格の統一に留め、**モジュールへの新機能追加はしない**。
- 名寄せレビュー画面 / CSV 統一の全モジュール展開 / マスタの事後マージ / プロジェクト・ダッシュボードは **Phase 2**。
- 構築順: 基盤 → todo → goals → wbs → skills → workload（§7.4）。

## 開発ワークフロー（必須ルール）

以下は**必ず守る**こと:

1. **作業は機能単位で Issue を立ててから始める。** 実装の前に GitHub Issue を作成し、何を作るか・受け入れ条件を明記する。
2. **作業はブランチを切って行い、PR を出す。** 直接 `main` にコミットしない。PR 本文に `Closes #<issue番号>` を入れて Issue と紐付ける。
3. **PR を出した後、必ず `/code-review` を実施する。** 指摘事項はマージ前に対応するか、対応しない場合は PR コメント or 別 Issue として残す。
4. **`/code-review` の指摘は次のサイクルで処理する。**
   1. 指摘内容を PR コメントとして残す（対応する／しないの方針も記載）。
   2. 対応すると判断した指摘を修正してコミット・プッシュする。
   3. 再度 `/code-review` を実施する。
   4. 新たな指摘がなくなるまで 1〜3 を繰り返す。問題がなければマージ確認に移る。

### 補足

- コミット・プッシュ・PR作成・マージは、ユーザーが明示的に指示したときのみ行う。
- コミットメッセージ・PR・Issue は日本語で記述する。
- モジュールの追加・修正は、着手前後に [`CONVENTIONS.md`](CONVENTIONS.md) §6 の完成チェックリストで自己点検する。
- テストは [`TESTING.md`](TESTING.md) に従う。ロジック変更後は `node test/run.js` を緑にし、変更したモジュール＋依存側（§2 マトリクス）を確認する。
- **ビルド不要・外部依存ゼロ・`file://` 動作・ローカル完結**を崩さない（ファイルは分割するが、ES Modules / `fetch()` を使わず classic script + `window.MK` で連携する）。
