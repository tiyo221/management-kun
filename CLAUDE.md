# CLAUDE.md

このリポジトリで作業する際の**運用ガイド**です。CLAUDE.md は日々の作業で常に守る手順・必須ルール（開発ワークフロー・運用ルール）を扱い、アーキテクチャ・共通データ規格・デザイン統一・フェーズなどの**詳細仕様は [`spec.md`](spec.md) を単一ソース（正）とする**。二重記載を避けるため、詳細は本書に書き写さず各仕様書へリンクする。記述が食い違ったら [`spec.md`](spec.md) を優先する。

## プロジェクト概要

**マネジメントくん** — **共通デザイン・共通データ規格・共通マスタ**のもとに、モジュールを追加・削除して育てていく管理ツール。

出発点として既存の5つの単一HTMLツール（目標みえるくん / スキル可視化 / タスク可視化 / todo-kun / WBS）を1つに統合したが、**統合自体が目的ではない**。統合はこのツールを立ち上げた経緯であり、ここから用途に応じてモジュールを増減させて発展させていくことが本来の狙い。

- 主要ファイル: [`index.html`](index.html)（シェル。これを開く）＋ `shared/*`（共有資産）＋ `modules/*`（各モジュール）
- 共通仕様: [`spec.md`](spec.md) — アーキテクチャ・共通データ規格・デザイン統一・フェーズの定義（**本書より spec.md を優先**）
- 共通マスタ: [`spec/masters.md`](spec/masters.md) — マスタ共通契約（§4.4.1）＋各マスタ（People / Project / Allocation / Product）の定義。マスタを追加・修正するときに読む
- 取込・移行・名寄せ: [`spec/import-migration.md`](spec/import-migration.md) — CSV 規約（§4.6）・旧データ移行（§7）・名寄せ（§8）。CSV/JSON 取込・移行・名寄せを触るときだけ読む
- モジュール統廃合: [`spec/consolidation.md`](spec/consolidation.md) — module→module 統合の移行テンプレート（手順書＋one-off スクリプト骨格＋テストパターン＋後始末）。Phase 3 の統廃合を実施するときだけ読む
- モジュール個別仕様: [`spec/modules/<id>.md`](spec/modules/) — 各モジュールの位置づけ・固有データ・CSV 列・旧データ移行（そのモジュールを直すときに読む）
- デザイン定義: [`DESIGN.md`](DESIGN.md) — Notion 風デザインシステム（トークン・コンポーネント）
- コーディング規約: [`CODING.md`](CODING.md) — 言語/実行環境・モダン JS・構造/命名・安全/堅牢性・オーバーエンジニアリング防止（**コードを書く前に参照**）
- 規約・チェックリスト: [`CONVENTIONS.md`](CONVENTIONS.md) — UI/レイアウト規約・モジュール分割（logic/view）・完成チェックリスト（**モジュール追加/修正時は必ず参照・自己点検**）
- データ永続化: ブラウザの `localStorage`（`mk:` プレフィックスで統一）。外部送信は行わない。
- 共有・バックアップ・AI 保守: JSON エクスポート / インポート（全体 / モジュール / 人 / プロジェクト単位）。一部モジュールは CSV にも対応（対象は [`spec.md`](spec.md) §5 の一覧表を正とする）。

目的（詳細は [`spec.md`](spec.md) §1）:

- 全モジュールでデザインを統一する（[`DESIGN.md`](DESIGN.md) の同一トークン・コンポーネントを使用）
- JSON / localStorage / CSV のデータ規格を1本化する
- 「メンバー（人）」「プロジェクト」を全モジュール横断のマスタとして共有する
- 共通部分を共有資産に切り出し、モジュールごとの重複実装をなくす（保守性）
- モジュールの追加・削除を低コストにする（共通基盤の上に載せ替える）

設計思想（重要）: **ビルド不要・外部依存ゼロ・`file://` 動作・ローカル完結**を崩さない。単一ファイルではなく **「単一ページ・シェル ＋ 共有資産 ＋ モジュール」** 構成とする（詳細な実装ルールは [`CODING.md`](CODING.md)）。

## 構成

仕様は2層＋トピック分離。全モジュールに効く**共通仕様**は [`spec.md`](spec.md)（§3.2 構成 / §4 データ規格 など。取込・移行・名寄せ ＝ §4.6・§7・§8 は [`spec/import-migration.md`](spec/import-migration.md) に分離）、**モジュール固有の仕様**は [`spec/modules/<id>.md`](spec/modules/)（位置づけ・固有データ・CSV 列・旧データ移行）。修正・機能追加時は「共通仕様＋対象モジュールの個別仕様」だけ読めば足りる。本書では重複させず、作業で常に守る**運用ルール**だけを挙げる:

- **logic/view を必ず分割する**。各モジュールは `MK.registerModule(id, def)` で登録し、シェルが `def.mount(container, ctx)` を呼ぶ（logic＝DOM 非依存の `MK.logic[id]`、view＝描画のみ。API 契約は [`spec.md`](spec.md) §3.5）。共有ヘルパ・追加手順・完成チェックリストは [`CONVENTIONS.md`](CONVENTIONS.md)。
- モジュールは**自分の名前空間 `mk:module:<id>` のみ**書き込む。マスタの変更は必ず `ctx.people` / `ctx.projects` 経由で行う（直接 localStorage を触らない）。

## コーディング規約

コードを書くときの規約（言語/実行環境・モダン JS・構造/命名・安全/堅牢性・オーバーエンジニアリング防止）は [`CODING.md`](CODING.md) に分離した。**コードを書く前に必ず参照する。**

要点だけ再掲すると: **ビルド不要・外部依存ゼロ・`file://` 動作・ローカル完結**を崩さない／ES Modules・`fetch()` を使わず classic script + `window.MK` で連携する／`var` 禁止・`===`/`!==`／DOM 出力は `textContent`・エスケープ（XSS 防止）／共通化は2か所以上で必要になってから（YAGNI）。

## 開発ワークフロー（必須ルール）

以下は**必ず守る**こと:

1. **作業は機能単位で Issue を立ててから始める。** 実装の前に GitHub Issue を作成し、何を作るか・受け入れ条件を明記する。Issue の書式は下記「Issue の書き方」に従う。
2. **作業はブランチを切って行い、PR を出す。** 直接 `main` にコミットしない。PR 本文に `Closes #<issue番号>` を入れて Issue と紐付ける。
3. **PR を出した後、必ず `/code-review` を実施する**（ユーザーがレビュー省略を指示したときはその限りでない）。指摘事項はマージ前に対応するか、対応しない場合は PR コメント or 別 Issue として残す。
4. **`/code-review` の指摘は次のサイクルで処理する。** 1巡は (a)〜(c) の3ステップからなる。
   - (a) 指摘内容を PR コメントとして残す（対応する／しないの方針も記載）。
   - (b) 対応すると判断した指摘を修正してコミット・プッシュする。
   - (c) 再度 `/code-review` を実施する。

   **ループの終了条件（必ず守る）:**
   - **自律的に回してよいのは3巡まで。その3巡は毎巡止まらず続けてよい。** 3巡を終えても **High / Medium** が残るなら「人間の確認が必要なレベル」と判断し、そこで止まって結果を要約して報告し、指示を仰ぐ。
   - **「3巡」は1回の許可あたりの予算であり、通算の上限ではない。** **「実装に入って」のような実装の指示が、最初の3巡ぶんの許可にあたる**（レビューを回すための追加の指示は要らない）。止まったあと再開の指示が出たら、そこから新たに3巡ぶん回してよい（最初から数えれば6巡目まで自動で進めてよい）。以後も同様に、許可のたびに3巡ずつ。
   - **予算になるのは「レビューの継続を含む指示」だけ**（「続けて」「もう一度回して」等）。**「この指摘だけ直して」のような個別の修正指示は予算にならない。** 直して報告し、そこで止まる（再レビューは回さない）。狭い指示を広い許可と読み替えない。
   - **High / Medium がゼロになったらそこが終点。** 3巡を使い切っていなくても打ち切り、マージ確認に移る。Low を潰しにいくために追加で回さない。修正は新しいコードを生み、それが次巡の新しい指摘を生む（レビュー → 修正 → コード増 → 指摘増）ため、ループに不動点は無い。
   - **「3巡」は巡回数であり、(a)〜(c) のステップ記号とは無関係。**

   **指摘への対応要否の基準:**
   - **Low の対応基準は「実害があるか」。** 実害のあるものだけ直し、実害のない将来的・理論的な指摘は今は対応せず、方針だけ PR コメントに残す。実際に問題が起きればそのとき改めて確認されるので、そのとき潰せばよい。**直した結果を確かめるための再レビューは回さない**（High / Medium がゼロなら上記のとおりそこが終点）。
   - **既出の指摘は既出として却下してよい。** `/code-review` は毎回コールドスタートで前巡の議論を知らないため、一度「対応しない」と判断した指摘を何度でも再提示する。前巡で結論が出ている指摘は、その旨をコメントに残して再検討しない。
5. **参照したドキュメントが実装・現状と食い違っていたら、そのまま放置しない。** 作業中に読んだ仕様・ガイド（`spec.md` / `spec/*` / `README.md` / `CONVENTIONS.md` / `TESTING.md` 等）が実装とずれていたら、軽微なら**同じブランチ／PR で更新する**。作業スコープを超える場合は `[core]`・`documentation` ラベルの Issue を立てて残す。ドキュメントの鮮度は「触った人がそのとき直す」ことで維持する（陳腐化の根本原因＝直し漏れを防ぐ）。
   - **spec のフェーズ項目・未決事項を完了させた PR は、該当記述を「実装済み」注記に書き換えるのではなく現在形へ刈る**（済んだ記述を積まない。取り消し線・「（#nn で〜した）」型の履歴注記も同様。履歴は git に任せる）。ただし「現在の判断の根拠」として参照される決定ログ（spec §9.6 の統廃合判断記録など）は履歴ではなく現行仕様の一部として残す。
   - **モジュールを追加・削除・改名したら、その id でドキュメント全体（`spec.md` / `spec/*` / `README.md` / `CONVENTIONS.md` / `TESTING.md` 等）を grep し、言及箇所を同期する**。「読んだ箇所だけ直す」では、そのセッションが開かなかったファイルの陳腐化を取り逃す（`daily` / `dashboard` 追加時に spec §1.4・§3.6 が古いまま残った実例＝#227）。**読んだ箇所を直すのではなく `grep <id>` で言及を探しに行く**。正準一覧（[`spec.md`](spec.md) §5）以外に id が散在していたら、その場で §5 参照へ寄せるか「例:」の少数へ刈り、同期対象そのものを減らす（網羅に見える列挙は、欠落が誤情報になる）。

### Issue の書き方

Issue は一覧から「どの領域か」「バグか機能追加か」がひと目で分かるように、かつ **Issue を立てるセッションと実装するセッションは別で、壁打ちの内容は実装側に引き継がれない**前提で Issue 単独で読んで実装できるように、次の書式で統一する。

**タイトル・ラベル**

- **タイトルは領域接頭辞で始める**: `[<領域>] <動詞から始まる要約>` の形にする。領域はそのモジュール／作業が属する棚（[`spec.md`](spec.md) §1.4 のゾーン）を示す。
  - 使う接頭辞: `[self]`（自分＝todo/goals）｜`[people]`（ピープル＝skills/resource/oneonone＋人マスタ）｜`[delivery]`（デリバリー＝wbs＋プロジェクトマスタ）｜`[product]`（プロダクト）｜`[tech]`（テクノロジー）｜`[core]`（共有基盤・シェル・仕様・テスト等、特定モジュールに属さないもの）。
  - 複数領域にまたがる場合は主たる領域1つを選ぶ（例: `[core]`）。
  - 例: `[people] スキル一覧の評価selectがtdに包まれず1列に寄る不具合を直す` / `[delivery] wbs を対象別 namespace へ移行する` / `[core] ヘッダ／ナビをモジュール増加に耐える構造へ見直す`。
- **種別は GitHub ラベルで示す**: 既存の標準ラベルを使う。バグ＝`bug`、機能追加・改善＝`enhancement`、ドキュメントのみ＝`documentation`。バグ Issue には必ず `bug` を付ける（タイトルにも「不具合」「バグ」等の語を入れて種別を明確にする）。ラベルは既存の標準ラベルのみを使い、新しいラベルはむやみに増やさない。

**本文の大項目（種別共通・この順で固定）**

| 大項目 | 中身 | 必須/任意 |
|---|---|---|
| `## 概要` | 何を・なぜ。1〜3行の要約 | 必須 |
| `## 背景` | **なぜ**（経緯・現状の課題）。別セッションが壁打ちを引き継がない前提で、単独で読んで文脈を掴める厚みで書く | 必須 |
| `## 詳細` | **何を**（やること）。バグは `### 再現手順`・`### 期待`・`### 実際` を立てる | 必須 |
| `## 決定事項` | 壁打ちで確定した方針・判断（採否理由込み）。無ければ「特になし」と書く | 必須（常設） |
| `## スコープ外` | 今回やらないこと | 任意 |
| `## 受け入れ条件` | 完了／修正の判定基準（チェックボックス `- [ ]`） | 必須 |
| `## 参考` | 関連 spec / Issue / PR | 任意（推奨） |

- **背景と詳細を混ぜない**: 背景＝「なぜ（経緯・課題）」、詳細＝「何を（やること）」で切り分ける。
- **決定事項は常設**: 壁打ちで固まったことを固着させる箱。実装セッションはその議論を知らないので、確定事項はここに残す。無ければ「特になし」と明記して未決を可視化する。

### PR の書き方

PR 本文も大項目を固定する（この順）。

| 大項目 | 中身 | 必須/任意 |
|---|---|---|
| `## 概要` | やったこと・なぜ | 必須 |
| `## 変更` | 具体的な変更点 | 必須 |
| `## テスト` | 実施内容と結果（`node test/run.js` の結果を含む） | 必須 |
| `## 受け入れ条件` | Issue のチェックリストを転記し、対応状況を反映（`- [x]`） | 必須 |
| `Closes #<issue番号>` | Issue 紐付け | 必須 |

- 導入部の見出しは Issue も PR も `## 概要` に統一する（「やりたいこと」「やったこと」は使わない）。変更点の節は `## 変更` に統一する（「変更内容」は使わない）。
- PR に `## 背景` は付けない（Issue 側で足りる。PR は差分の説明に集中する）。

### 大項目を埋めるときのルール（推測で埋めない／勝手に決めない）

- **必須の大項目は必ず立てる。埋められない場合は省略・推測せず、ユーザーに確認する。** 特に `## 背景` が薄いまま Issue を立てない。
- **実装時に Issue の `決定事項` に無い設計判断が必要になったら、勝手に決めず着手前にユーザーへ確認する。**「決まっていないから実装できない」ときは推測で進めず質問する。

### 補足

- **止まる境目は「元に戻しやすいか」で引く。不可逆なのはマージだけなので、明示的な指示を待つのもマージだけ。** ブランチ作成・コミット・プッシュ・PR作成は実装の指示に含まれるものとして進めてよく、1つずつ許可を取らない。`/code-review` をどこまで回してよいか・どこで止まって報告するかは、必須ルール 4 の「ループの終了条件」に従う（ここでは繰り返さない）。
- コミットメッセージ・PR・Issue は日本語で記述する。
- モジュールの追加・修正は、着手前後に [`CONVENTIONS.md`](CONVENTIONS.md) §6 の完成チェックリストで自己点検する。
- テストは [`TESTING.md`](TESTING.md) に従う。ロジック変更後は `node test/run.js` を緑にし、変更したモジュール＋依存側（§2 マトリクス）を確認する。
- コードを書くときの規約・禁止事項は [`CODING.md`](CODING.md) を参照する（**ビルド不要・外部依存ゼロ・`file://` 動作・ローカル完結**を崩さない）。
